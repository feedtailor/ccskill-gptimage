## クローズ理由(2026-06-11、#020 を踏まえた再吟味)

本 issue は #020(install.sh + 最小ディスパッチャ)の実装により **主目的が達成済み** となり、かつ
**当初設計が ccskill-gptimage の性質に適合していなかった** ため、doctor 構想も含めてクローズする。

### 根本: gmail はステートフル、gptimage はステートレス

当初設計は ccskill-gmail のアーキテクチャ(`commands/` + `lib/` + `registry` + `update-all` +
`register` + プロジェクト単位 install)をほぼ写経していた。だがその機構が gmail で必要なのは、
**gmail がプロジェクトごとに状態を持つ** から:

- gmail: 各プロジェクト配下に `.ccskill-gmail/`(config.js / api / binding.json / GAS デプロイ ID /
  アカウント紐付け)という実体を置く。ゆえに「どこに install したか」を registry で追跡し、
  update-all で全箇所へ配り直し、register で既存を取り込み、doctor で各プロジェクトの状態を診断する
  必要がある。
- gptimage: **プロジェクト配下に一切状態を置かない(完全ステートレス)**。machine-global な状態
  (Codex 認証 `~/.codex/auth.json` / `OPENAI_API_KEY` / repo の `.env`)だけ。

したがって registry / update-all / register / プロジェクト単位 install は
**追跡すべき対象が無いのに用意する空回りの機構** であり、本スキルには過剰だった。

### #020 が代替した内容

#020 は「ユーザレベル symlink 1 本 + `git pull`」モデルで Phase 1 の主目的を必要十分に達成した:

- `setup`(venv + pip)→ install.sh Step 1-2 で実装済み
- `install`(プロジェクト単位 symlink)→ **ユーザレベル登録(`~/.claude/skills`)に置換**。全プロジェクトから
  自動認識され、プロジェクト単位 install はむしろ不要(symlink を撒くアンチパターン)
- `uninstall` → dispatcher `uninstall`(ユーザレベル)で実装済み
- `update` / `update-all` → **`git pull` が更新そのもの**。複数 install も registry も存在しないため不要
- `register` → registry が無いため不要
- ディスパッチャ本体(`ccskill-gptimage <subcommand>`)→ #020 で実装(`generate` / `uninstall` /
  `version` / `help`)
- SKILL.md/ja.md + prompts/ の呼び出し例をディスパッチャ経由(`ccskill-gptimage generate`)へ統一済み

### 恒久的に不採用とするもの

- `registry.sh`(.registry.json)/ `update-all` / `register` / プロジェクト単位 `install` / `release`(配布 zip)
  — ステートレスな本スキルには対象が無く、配布は `git clone` で足りる。

### 後続 issue へ移譲

- `doctor`(API キー / Org Verification / 依存の診断)→ **#021 として別途起票**。診断はコスト UX とは
  関心が異なるため独立 issue とした。
- `info costs`(size×quality コストマトリクス表示)→ #010(コスト見積り警告 + モデレーション UX)の中で実装。

いずれも #020 の dispatcher が case 文への追加だけで `doctor` / `info` を生やせる構造になっている。

### 依存関係の更新

- 当初 `blocks: [009, 010]` だったが、#009/#010 はいずれも `generate_image.py` の CLI 機能であり、
  dispatcher が既に #020 で存在するため **#020 で実質ブロック解除済み**。両 issue の `depends_on` から
  008 を除去した。

## 当初設計(参考 — ccskill-gmail 機械移植・不採用)

> 以下は起票時の設計。上記理由により採用しない。記録として残す。

ccskill-gmail のアーキテクチャに合わせて CLI ディスパッチャを導入。`ccskill-gptimage <subcommand>` 形式で setup/install/info/doctor/status/update/update-all/uninstall/register/release を提供する想定だった。

```
ccskill-gptimage/
├── ccskill-gptimage          # ルート直下のディスパッチャ(bash)  ← #020 で最小版を実装
├── commands/
│   ├── setup.sh              # venv 作成 + pip install            ← #020 install.sh に統合
│   ├── install.sh            # 別プロジェクトへのインストール       ← ユーザレベル登録に置換
│   ├── uninstall.sh          #                                    ← #020 dispatcher に統合
│   ├── update.sh             # ディレクトリ単位の更新              ← git pull に置換(不要)
│   ├── update-all.sh         # registry に基づく一括更新           ← 不採用
│   ├── info.sh               # 環境/設定情報の表示                 ← #010 へ(info costs)
│   ├── status.sh             # 現状診断                           ← 不採用(doctor と重複)
│   ├── doctor.sh             # セットアップ問題のスキャン           ← #021 として独立
│   ├── register.sh           # 既存インストールを registry に登録   ← 不採用
│   └── release.sh            #                                    ← 不採用
└── lib/
    ├── registry.sh           # .registry.json 操作                ← 不採用
    ├── update_check.sh       # GitHub から新版チェック             ← 不採用(将来別 issue で再検討可)
    └── permissions.sh        #                                    ← 不採用
```
